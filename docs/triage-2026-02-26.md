# Triage zgłoszeń (2026-02-26)

## [1] ℹ The configuration schema version does not match the CLI version 2.4.4 - należy poprawić konfigurację aby działała dla 2.4.4

- **Zasadność:** **Potwierdzony**. W `biome.json` repo wskazuje schemat `2.4.3`, podczas gdy `package.json` ma `@biomejs/biome` w wersji `^2.4.4`, więc metadane konfiguracji są niespójne i mogą dawać błędne podpowiedzi walidacyjne w edytorach.
- **Decyzja:** **SHOULD**
- **Priorytet:** **Medium**
- **Wpływ:** utrzymanie OSS, spójność narzędzi developerskich (niski wpływ runtime dla użytkowników końcowych).
- **Ryzyko regresji:** **Low** — zmiana dotyczy URI schematu i nie wpływa na logikę pluginu.
- **Jak zweryfikować:**
  1. Otworzyć `biome.json` i sprawdzić `$schema`.
  2. Porównać z wersją `@biomejs/biome` w `package.json` i lockfile.
  3. Uruchomić `npm run lint`, upewnić się, że config jest akceptowany.
- **Jeśli naprawiać:** Podnieść `$schema` do `https://biomejs.dev/schemas/2.4.4/schema.json`, następnie uruchomić lint/check i upewnić się, że nie doszły nowe reguły wymagające dodatkowych zmian.

## [2] Bump Biome schema to v2.4.4 in biome.json

- **Zasadność:** **Potwierdzony**. To samo zagadnienie co [1], ale jako konkretne zalecenie techniczne; obecny plik ma nadal `2.4.3`.

- **Priorytet:** **Medium**

## Podsumowanie

### Top 5 rzeczy do naprawienia natychmiast
1. Rozjazd `package.json` vs `package-lock.json` blokujący `npm ci` (**Blocker**).
2. Synchronizacja wersji `@biomejs/biome` w lockfile z deklaracją w `package.json` (element blockera).
3. Synchronizacja `@types/node` między lockfile i manifestem (element blockera).
4. Aktualizacja `$schema` Biome do 2.4.4, aby uniknąć dryfu narzędziowego.
5. Dodanie prostego checka w CI, który wyłapie dryf lockfile wcześniej (np. dedykowany job `npm ci` przed build/test).

> Uwaga: z dostarczonej listy tylko pozycja [3] jest blockerem wydania; pozostałe to zadania jakościowe.

### Top 5 rzeczy do odłożenia
1. Rozdzielanie zmian kosmetycznych configu od zmian zależności, jeśli zespół chce minimalnych PR-ów.
2. Ewentualne dalsze podbicia narzędzi deweloperskich ponad 2.4.4 (po ustabilizowaniu CI).
3. Refaktoryzacje skryptów release — dopiero po naprawie lockfile.
4. Dodatkowe uspójnienia stylu/lintu niezwiązane z awarią CI.
5. Rozbudowa dokumentacji procesu wersjonowania, jeśli zasoby są ograniczone.

### Rzeczy do odrzucenia (WON’T)
- Brak pozycji kwalifikujących się do **WON’T** w dostarczonym zestawie 3 zgłoszeń.

### Ryzyka wprowadzania zmian
- Największe ryzyko regresji jest przy regeneracji lockfile (możliwa zmiana transitywnych zależności).
- Niskie ryzyko przy samej zmianie `$schema` w `biome.json`.

### Sugestia kolejności prac (5–10 kroków)
1. Naprawić lockfile (`npm install`) i sprawdzić diff zależności.
2. Uruchomić `npm ci` lokalnie jako test bramkowy.
3. Uruchomić `npm run check` (lint + typecheck + test).
4. Zaktualizować `biome.json` `$schema` do 2.4.4.
5. Ponownie uruchomić lint/check.
6. Otworzyć mały PR z jasnym opisem „dependency sync + schema sync”.
7. Po merge dodać/przesunąć w CI krok `npm ci` na początek pipeline.
